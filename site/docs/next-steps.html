Your first conversation

You have a scaffolded project open in Claude Code. Now what? A few prompts that get great results, plus what to expect back.

i

Claude Code's job is not to guess what you want. Tell it plainly. Short, specific prompts beat long vague ones. Saying "make me a website" sends Claude in a direction that's almost certainly not yours.

Three good first prompts

1. Audit what you have

Read CLAUDE.md and tell me, in plain English, what this project is
configured for and what's still TBD. Don't change anything yet.

Claude will read your CLAUDE.md, summarize the stack and discipline, and list anything still blank. This is a cheap way to verify the scaffold worked and to see what Claude understands about your project.

2. Write your first proposal

I want to build a [thing]. Help me write a proposal for it in
docs/proposals/ following the TEMPLATE.md format. Ask me clarifying
questions one at a time before writing.

The scaffolded project comes with a docs/proposals/ folder and a template. Writing a proposal first — even a short one — captures the design decisions before code happens. Future-you (and future-Claude) will thank you.

3. Plan the first feature

Now make a plan in docs/plans/ dated today, for shipping the first
working version of [thing]. Use the TEMPLATE.md format. Keep action
items atomic — one PR worth each.

Plans are time-bounded execution lists. They live in docs/plans/ and are dated like 2026-05-17-first-version.md. Atomic action items let Claude (or you) work through them one at a time without drift.

What Claude knows by default

Because CLAUDE.md is loaded automatically, Claude already knows:

  • Your project's tech stack (whatever you filled in for Frontend / Backend / DB).
  • The four-layer memory hierarchy — memory, proposals, plans, git history.
  • Token discipline — to use semantic search instead of reading whole files when possible.
  • Voice rules — describe what is, not what it isn't; no marketing language.
  • Commit and PR discipline — small PRs, cite proposals, never --no-verify.

You don't have to repeat any of this in your prompts. Claude already has it.

The skills Claude can reach for

Your CLAUDE.md lists Claude Code skills that fit your project. Some examples:

  • concise-planning — produces atomic checklists with very few clarifying questions.
  • code-reviewer — runs before you merge.
  • ui-ux-pro-max (UI variant) — 99 UX guidelines applied to UI work.
  • claude-api (agent variant) — for building apps that call the Claude API.

You don't have to know skill names to use them — Claude reaches for them automatically when the task fits. If a skill isn't installed, Claude will tell you, and you can install it with a single command.

The memory layer that grows with you

As you work, Claude writes "auto-memory" notes to a folder at ~/.claude/projects/<your-project>/memory/. This is permanent across sessions. Examples of what gets saved:

  • Your role and background, if you mention it.
  • Feedback you've given ("don't use mocks in tests", "prefer Bash over PowerShell").
  • Project-level facts and deadlines.

Every future session loads this memory automatically. Over weeks, Claude accumulates context about how you work — without you re-explaining.

!

Ask Claude what it remembers at any time:

What do you have in memory about me and this project?

It'll show you the saved files. You can also tell it to forget specific things if they're outdated.

When to ask Claude to plan vs. just code

For anything beyond a tiny fix: ask for a plan first. Even one sentence: 

Before you make any changes, write a one-paragraph plan: what you're
going to change, in which files, and why.

This catches misunderstandings before Claude touches code. You'll redirect the plan once and save yourself untangling unwanted changes.

How to commit your changes

When you and Claude have made a change you're happy with, save it with git. In Claude Code, just ask: 

Commit the changes you just made with a clear commit message.

Claude will run git add and git commit for you. Then in GitHub Desktop, click Push origin to send the commit up to GitHub.

When to start a new session

Long conversations get expensive in tokens and Claude can lose focus. Rough rule: end the session when the current task is done, and start a new one for the next task. Auto-memory persists across sessions, so you don't lose context — just the back-and-forth of that specific conversation.
