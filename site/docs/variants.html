What's actually in each variant?

When you scaffold a project you pick UI App or Agent App. Both variants share the same foundation — they differ in which extras they layer on top. This page explains every extra in plain language, what it does, and when it kicks in.

First — what's a "skill"?

Most of the extras below are called "skills." A skill is a markdown file stored at ~/.claude/skills/ on your computer. It tells Claude how to handle a specific kind of task — writing UI, planning, reviewing, etc. Skills are installed once per machine, then any project can use them.

The scaffolded project doesn't install the skills — it just references them in CLAUDE.md so Claude knows to reach for them when relevant. If a skill isn't installed when Claude tries to use it, you'll be told, and you can install it with a single command.

i

A few extras aren't skills — they're files in your project (like UX_CONTRACT.md) or design principles baked into the CLAUDE.md template. Each entry below says which kind it is.

The shared foundation

Both variants ship with this set — they don't appear in the variant breakdowns because they're always there. Listed here for completeness.

  • Memory hierarchy — files for proposals (architectural decisions), plans (in-flight work), test runs (real journey logs), and auto-memory (persistent across sessions). Documented in docs/AGENTS.md.
  • Three MCP servers pinned — Serena (semantic code search), Context7 (live library docs), GitHub (issue/PR access).
  • Token discipline rules — guidance baked into CLAUDE.md telling Claude to use semantic search instead of reading whole files, never re-read what's loaded, etc.
  • Voice rules — describe what is, not what it isn't; no marketing language; no self-congratulation.
  • Commit + PR discipline — small PRs, cite proposals, never --no-verify.

UI App

scaffolder flag: --type ui-app · for websites, dashboards, marketing pages

Layered on top of the shared foundation are three groups of extras focused on building good user interfaces.

Group 1 of 3

A visual design rulebook

Specific rules for layout, color, animation, and accessibility — plus a contract file your project's PRs cite. Three pieces work together.

A 99-rule design checklist

skill · ui-ux-pro-max

A curated checklist of 99 specific UI/UX rules synthesized from industry best practices — color contrast minimums, button-size guidelines, animation timing windows, anti-patterns that the skill will actively refuse to produce. It also ships ~160 color palette recommendations.

When it kicks in: any time you ask Claude to build, review, or critique a visual interface. Saying "make this card more readable" triggers it; saying "rename this function" doesn't.

Design-system structure

skill · design-system

A pattern for organizing visual values (colors, spacing, font sizes) in three layers:

  • Primitive — raw values like #7dd3fc or 16px.
  • Semantic — what they mean: "primary action color," "card padding."
  • Component — what uses them: "button border," "modal background."

This way, changing one primitive (the cyan accent, say) ripples through everywhere it's used without touching component code.

When it kicks in: when you set up the project's styling or add a new visual concept. Useful even for small projects — it's the difference between "edit 40 files to change the accent color" and "edit one variable."

Your project's UI rulebook

file · docs/UX_CONTRACT.md

Unlike the other entries, this one is a file inside your project, not a global skill. It's a 12-point checklist your PRs reference and cite. The scaffolded version covers:

  • The master interaction contract (attention → question → commitment → feedback → reveal).
  • Color semantics (six tokens, one meaning each).
  • Animation budget (150–300ms, animate transform/opacity only, respect prefers-reduced-motion).
  • Navigation rules (one concern per nav item, no stub status items).
  • Accessibility minimums (4.5:1 contrast, focus rings, aria labels, keyboard nav).
  • Touch targets (≥44×44px, ≥8px spacing).
  • A 12-item review checklist every UI PR must pass.

You can edit, extend, or replace it — it's your project's rules, not a mandate. The point is having some contract in writing so visual discipline doesn't drift.

When it kicks in: at PR-review time, and when Claude is about to make a visual change. The file is referenced from CLAUDE.md so it's part of every session's context.

Group 2 of 3

First-use onboarding patterns

A behavioral-research framework for taking new users from curious to committed. One piece.

Onboarding playbook

skill · onboarding-psychologist

A framework for designing first-time-user experiences based on behavioral research. It uses the IDENTITY-TO-HABIT arc: define the first win → remove unnecessary setup → create ownership through a small meaningful action → attach a stable cue (link to existing routine) → reinforce identity (reflect the user as someone who succeeds at this).

It explicitly rules out feature tours, "12 things to know" walls, and empty-state pages that show CRUD forms instead of value.

When it kicks in: when you're designing the very first thing a new user sees and does. Especially valuable for products where adoption is a real concern.

Group 3 of 3

Anthropic's frontend plugin

General-purpose assistance for component design and visual decisions. One piece.

Frontend design helper

plugin · frontend-design (Anthropic official)

Anthropic's officially maintained frontend design plugin. General-purpose companion for component design and visual decisions — less opinionated than ui-ux-pro-max, more focused on Anthropic's idea of frontend best practices.

When it kicks in: alongside the other UI skills, for general-purpose frontend work. The two skills complement each other rather than overlap.

Agent App

scaffolder flag: --type agent-app · for LLM apps and AI agents

Layered on top of the shared foundation are three groups of extras focused on building applications that themselves call LLMs.

Group 1 of 3

Agent architecture & orchestration

How to structure single-agent or multi-agent systems — handoffs, supervision, parallel work. Two pieces.

Agent architecture guide

skill · ai-agents-architect

Helps you think through how your app should be structured. Key questions: do you need a single agent (one personality, one set of tools)? Multiple agents (specialized roles talking to each other)? An orchestrator (a top-level agent that delegates to sub-agents)? It walks you through the tradeoffs based on what your app actually has to do.

When it kicks in: at the start of the project, and again any time you're tempted to "just add another agent." Used early, it prevents the most common agent-app architectural mistakes.

Multi-agent coordination patterns

skill · agent-orchestrator

If your app uses more than one agent, this covers how to get them to cooperate without stepping on each other. Patterns include:

  • Sequential handoffs — agent A finishes, hands the conversation to agent B.
  • Supervisor / worker — one agent delegates tasks to specialists.
  • Parallel work — multiple agents tackle independent parts of a problem at the same time.
  • Conflict resolution — what happens when two agents disagree about state.

When it kicks in: only relevant if your app has multiple agents. Single-agent apps can ignore this entirely.

Group 2 of 3

Memory between sessions

Patterns for what your agent remembers and how — vector recall, structured facts, or both. One piece.

Memory system design

skill · agent-memory-systems

Patterns for how your agent remembers things between conversations. Most agent apps need some persistence — what the user has said before, what they prefer, what the agent has learned. This skill covers the practical options:

  • Vector databases (Pinecone, LanceDB, pgvector) for fuzzy semantic recall — "what did we talk about regarding logistics?"
  • Structured databases (Postgres, SQLite) for facts and structured state — "user prefers metric units."
  • When to use which, and how to mix them.

When it kicks in: when you're designing the persistence layer — usually right after the architecture is decided. Get this wrong and you'll be re-architecting later.

Group 3 of 3

Building on the Claude API

Anthropic's official SDK skill plus a portable-identity rule so user data survives if you swap LLMs later. Two pieces.

Anthropic SDK helper

skill · claude-api (Anthropic official)

Anthropic's official skill for building apps that call Claude. It covers the API itself, plus advanced features:

  • Prompt caching — make repeat calls much cheaper.
  • Tool use — let Claude call your functions.
  • Streaming — show output as Claude generates it.
  • Batching — process many requests at once at lower cost.
  • Model migration — when Anthropic releases a new model, this skill helps you switch cleanly.

It auto-loads any time your code imports anthropic or @anthropic-ai/sdk.

When it kicks in: any time you're writing code that calls Claude's API. The auto-load means you usually don't have to ask for it explicitly.

Portable-identity design rule

discipline · documented in CLAUDE.md

Not a skill — a design principle baked into the variant's CLAUDE.md. It says: your app's user data (memory, skills, preferences, integrations) should be decoupled from the underlying LLM so you can swap models without losing what the app knows about its users.

Concretely:

  • The LLM choice lives in config (e.g. MODEL=claude-opus-4-7), not in your domain code.
  • User memory persists in your own DB, not in the model's context window.
  • Skills and tools are defined in your project, not in model-specific format.
  • Export/import are first-class — you can pull a user's identity bundle out.

Even if you don't plan to swap LLMs, designing as if you might forces cleaner separation that pays off everywhere.

When it kicks in: at schema-design time. Schema decisions answer "is this portable?" before "is this convenient?"

How to add or remove pieces

Variants aren't sacred. After scaffolding, every piece is just a file you can edit:

  • Add a skill to a variant for all future projects → edit templates/<variant>/CLAUDE.md.extension in this repo.
  • Add a skill to your specific project → edit your project's CLAUDE.md directly.
  • Remove the UX contract → delete docs/UX_CONTRACT.md in your project (or just stop citing it).
  • Add a new variant entirely → see CONTRIBUTING.md in the main repo for the recipe.

How Claude reaches for skills

You don't have to memorize skill names. Claude has the list (from CLAUDE.md) and reaches for them when the task fits. You can always ask explicitly — "use concise-planning for this" — but usually you just describe what you want and Claude picks the right tool.

If a skill isn't installed yet, Claude will tell you and give you the install command. Skills install at the user level (one-time per machine), so once you've installed one for a project it's available everywhere.
