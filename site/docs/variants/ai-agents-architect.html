Agent architecture guide

A decision-framework skill for autonomous-agent architecture. Helps you pick between single-agent and multi-agent designs, between ReAct loops and Plan-and-Execute, between adding an orchestrator now or later.

Heads-up: The exact slug ai-agents-architect isn't a definitive single public package. The patterns it documents are well-established in the agent-systems literature; the framework below is what any skill under this name should cover, drawn from public Anthropic guidance and the agent-systems research literature.

The first decision: how much autonomy?

Before picking architecture, decide where your agent sits on the autonomy spectrum:

  • Suggester — proposes actions, user approves each one. Lowest autonomy, lowest risk.
  • Assistant — takes action within tight rails, reports back. Most current AI apps live here.
  • Operator — takes multi-step action with checkpoints. Needs robust error handling.
  • Autonomous — runs unsupervised on goals, only escalates exceptions. Needs hard guardrails.

Each level needs different architecture. A suggester can be a stateless single call; an autonomous agent needs memory, retry logic, escalation paths, and observability.

Pattern 1: ReAct (Reason + Act)

The simplest agent loop. On each turn:

  1. Reason — think about what to do next given the goal and current state.
  2. Act — call a tool.
  3. Observe — read the result.
  4. Repeat until the goal is met or the loop budget is exhausted.

Good for: tasks where the right next step depends on what just happened. The agent can correct mid-flight.

Watch out for: infinite loops, getting stuck on sub-problems. Cap the iteration count and add an escape valve.

Pattern 2: Plan-and-Execute

Two-phase. First the agent produces a complete plan (a list of steps). Then it executes the plan, one step at a time. If a step fails, optionally re-plan.

Good for: tasks with a knowable shape — research jobs, data pipelines, multi-step transformations. The plan is auditable before execution.

Watch out for: brittle plans. Real-world conditions drift; if your agent can't re-plan mid-execution, it'll execute a stale plan against a changed world.

Pattern 3: Tree-of-Thoughts / Explore-and-Backtrack

For tasks where you'd want a human to consider multiple approaches before committing. The agent explores several possible action branches, evaluates each, picks the best.

Good for: creative problems, puzzles, hard reasoning.

Watch out for: token cost. Each branch is its own reasoning trace.

Single agent vs. multi-agent

The skill's most important decision tree. Defaults:

Use a single agent when…

  • The task fits in one mental model (one persona, one set of tools).
  • You can describe everything the agent does in one paragraph.
  • State and context flow linearly.
  • The cost of "what does this agent do?" is low enough to skip specialization.

Add a second agent when…

  • The personas are genuinely different (researcher vs. writer; reviewer vs. author).
  • Context for one role would pollute the other's reasoning.
  • You want to run two perspectives in parallel and compare outputs.
  • One role needs a much larger or much smaller context window than the other.

The trap: splitting an agent because "more agents = more sophisticated." Multi-agent systems pay for themselves only when the split has a real reason. Otherwise you've added handoffs and shared-state bugs to fix what wasn't broken.

When to introduce an orchestrator

An orchestrator is a top-level agent that doesn't do the work itself — it delegates to specialist sub-agents and merges results. Add one when:

  • You have ≥3 specialist agents and the routing logic is non-trivial.
  • You need to run sub-agents in parallel.
  • You need to enforce policies (cost limits, retries, fallbacks) across all sub-agents.
  • You need a single point of observability.

For 1-2 specialists, the calling code can usually orchestrate. An explicit orchestrator pays its weight at 3+.

Common anti-patterns the skill warns against

  • The infinite tool loop. No iteration cap. Agent keeps reasoning, never converges.
  • Tool overload. Giving the agent 40 tools when 6 would do. Choice paralysis happens to LLMs too.
  • Vague success criteria. "Help the user" is not a goal. "Return a JSON object matching this schema" is.
  • Memory as an afterthought. Stateless agents pretend to remember by re-reading history every turn. Token-expensive and brittle.
  • Multi-agent for its own sake. See above.

When it kicks in

Activates on prompts about agent design, autonomy boundaries, or orchestration choices. Examples:

  • "Should this be one agent or multiple?"
  • "How do I structure an agent that can both research and write?"
  • "My agent is getting stuck in loops — what's wrong with the architecture?"
  • "When should I add an orchestrator?"
