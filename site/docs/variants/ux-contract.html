Your project's UI rulebook

Unlike the other UI App entries, this one is a file inside your project, not a global skill. It's a 12-point checklist your visual PRs cite. The scaffolded version sets sensible defaults; you edit it to match your project's specific rules.

What's in the scaffolded version

The template ships a contract that's been refined across real projects. You can keep it, extend it, or rewrite it — but starting from this base is faster than from scratch.

Rule zero — the master interaction contract

Attention → Question → Commitment → Feedback → Reveal. The user must commit to a decision before the system reveals structure. No auto-advance. No answer-before-input. No forward navigation during reasoning. Every other rule serves this one.

Onboarding arc — IDENTITY-TO-HABIT

How a user gets from outsider to participant. Five steps:

  1. Define the first win — the smallest meaningful success.
  2. Remove unnecessary setup — minimize early decisions.
  3. Create ownership through action — small task that creates investment.
  4. Attach a stable cue — link return-visits to existing routines.
  5. Reinforce identity — reflect the user as someone who succeeds.

Two-language coherence

When the user commits to something in one representation, the paired representation updates at the same time. If a paired representation doesn't exist, say so explicitly — silence reads as broken.

Empty vs filled — ? slots and fill-pop cards

Empty values render as dashed-border ? placeholders. Filled values render as solid cards with a fill-pop entry animation. The contrast is intentional and load-bearing — users can tell at a glance whether they're making progress.

Hand-crafted, attempt-aware feedback

  • Attempt 1: Minimal — "not quite, try another."
  • Attempt 2: Same minimal nudge with slight rewording.
  • Attempt 3: Actual hint — concrete pointer.
  • Attempt 4+: "Show reasoning" — explains the rule but requires the user to still apply it.

Never auto-generated from a model at runtime without the per-attempt structure.

Wrong-but-valid ≠ wrong-and-invalid

  • Suboptimal but valid: amber tint, two buttons — "Use this anyway" and "Choose again". Log the suboptimal pick; debrief later.
  • Wrong and invalid: red flash, block. Forces retry.

Mixing them confuses users about what's permitted.

Color semantics — six tokens, one meaning each

The contract defines a small fixed set of color tokens. Don't introduce new colors for new states.

  • Attention target — neutral border with subtle pulse
  • Candidate (selected, unjudged) — amber tint, amber border
  • Correct / live — emerald tint, emerald border
  • Active / pivot — blue tint, blue border
  • Invalid — red flash, red border
  • Skipped / dimmed — reduced opacity

Animation budget

  • Duration: 150–300ms for state changes.
  • Spatial continuity: animate the path between two places.
  • Respect prefers-reduced-motion.
  • Never animate width/height — animate transform and opacity only (GPU-accelerated).

Navigation rules

  • One nav item per concern. If two pages answer the same question, they belong on one page.
  • The sidebar is not a TODO list. Stub status items are forbidden — either build the surface or remove the link.
  • Every page should feel like the same product.

Voice and tone

  • Describe what is, not what it isn't.
  • No marketing language ("the unlock," "delightful," "transformative").
  • No self-congratulation.
  • No defensive contrasts ("real X not Y").
  • No conversation-language in product surfaces.

Accessibility — non-negotiable

  • Color contrast ≥ 4.5:1 normal text, ≥ 3:1 large.
  • Focus rings visible (2–4px) on every interactive element.
  • Aria labels on icon-only buttons.
  • Keyboard navigation matches visual order.
  • prefers-reduced-motion respected.

Touch & interaction — non-negotiable

  • Minimum 44×44px touch targets, 8px+ spacing between.
  • Loading feedback within 100ms of every action.
  • No hover-only affordances.
  • No 0ms state changes — at minimum 150ms easing.

The 12-point review checklist

Every UI PR must pass this checklist or explicitly call out the trade-off:

  1. Master contract — does the user commit before the system reveals?
  2. Forward navigation locked during reasoning?
  3. Paired representation present, or its absence explicitly named?
  4. Empty vs filled visually distinct?
  5. Hints attempt-aware, or scripted placeholder explicitly labeled?
  6. Wrong choices distinguished invalid (red, block) vs suboptimal (amber, dual-button)?
  7. Color semantics consistent with the table above?
  8. Animations purposeful, within budget, respect reduced-motion?
  9. Accessibility — contrast, focus rings, aria labels, keyboard?
  10. Touch targets 44px+, spacing 8px+, loading feedback within 100ms?
  11. Voice clean — no marketing language?
  12. Sidebar does not gain a stub?

How to use it in practice

Cite it in PR descriptions:

## Test plan
- [x] UX_CONTRACT items 7-10 pass
- [x] Focus rings visible on all new buttons
- [ ] Touch target audit not done (mobile out of scope)

Claude reads UX_CONTRACT.md as part of every session because it's linked from CLAUDE.md. When you ask for a visual change, Claude checks against the contract before suggesting code.

Customizing it

You should edit it. The scaffolded version is a sensible default; your project will have its own constraints. Examples of edits that make sense:

  • Replace the color semantics table with your brand palette.
  • Add platform-specific rules (a desktop app might have different touch-target rules than a mobile site).
  • Add rules for specific domains — accessibility for healthcare, regulatory copy for finance.
  • Remove the onboarding arc if your product isn't user-facing.

The point isn't this specific contract — it's having some contract in writing so visual discipline doesn't drift.
