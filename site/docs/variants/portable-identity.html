Portable-identity design rule

Not a skill — a design principle baked into the Agent App variant's CLAUDE.md. Your app's user data — memory, skills, preferences, integrations — should be decoupled from the underlying LLM so you can swap models without losing what the app knows about its users.

Type
Discipline (documented in CLAUDE.md)
Maintainer
local Your project's discipline — edit freely
Triggers
Schema-design decisions, LLM choice, data storage
Install
Ships with agent-app scaffolds automatically

The principle

Treat the LLM as a replaceable component. The app's value lives in the user data, the skills, the integrations, and the workflows — not in the choice of model. When a better, cheaper, or faster model appears (and they appear quickly), switching shouldn't lose what your users have built up.

Even if you have no intention of switching LLMs, designing as if you might forces a cleaner architecture that pays off everywhere — easier testing, easier multi-tenancy, easier compliance.

What the rule says, concretely

1. LLM choice lives in config, not in your domain code

Wrong: 

# domain logic with the model baked in
def summarize_user_journal(journal_text):
    response = anthropic.messages.create(
        model="claude-opus-4-7",
        max_tokens=1024,
        messages=[{"role": "user", "content": journal_text}]
    )
    return response.content[0].text

Right: 

# domain logic that asks "the LLM" — config decides which
def summarize_user_journal(journal_text):
    return llm.summarize(journal_text, max_tokens=1024)

# config.py
MODEL = os.environ.get("MODEL", "claude-opus-4-7")
llm = LLMClient(model=MODEL)

2. User memory persists in your own database

Don't rely on conversation history living in the model's context window as your only persistence. If you swap models tomorrow, that history vanishes. Persist memory in your own DB (vector store + structured), and load relevant slices into context per request.

This is also what makes memory systems a separate piece — the rule is "own your memory layer."

3. Skills and tools are defined in your project, not in model-specific format

Tool schemas, skill files, integration definitions — store them in your project. Adapt to whatever shape each model wants at call time.

The Anthropic SDK has one tool schema shape; OpenAI has another; Gemini's tools look different again. Your skill definitions shouldn't know which one you're using. Have an adapter layer that converts your canonical format to whatever the active LLM expects.

4. Export/import is first-class

Build /export and /import endpoints from day one. Your users should be able to:

  • Pull their entire identity bundle (memory, preferences, skills, integrations) out as JSON.
  • Reimport it into a fresh instance, possibly running on a different LLM.

This is the test of whether you've actually achieved portability. If export/import roundtrips losslessly, your data is genuinely LLM-independent. If it doesn't, something's coupled that shouldn't be.

Why this matters more than people expect

  • Model churn is fast. The frontier model two years from now will not be the one you started on. Apps that lock in to a specific model become legacy quickly.
  • Multi-tenancy demands it. If different customers want different models (cost, region, compliance), your app needs to swap per request.
  • Testing benefits. Mock LLMs are trivial when your domain code asks "the LLM" instead of a specific provider.
  • Compliance benefits. Some regulated industries require the ability to migrate off specific vendors. Having that capability in your design from day one avoids painful retrofits.
  • The cost story changes. Today's most expensive model is sometimes tomorrow's mid-tier. Apps with portable identity can downgrade automatically as cheaper models become capable enough.

Anti-patterns

  • Storing the model name on user records. Tells you which model was active when this user signed up — but nothing else useful. Better: store the data, let config decide the model.
  • Using model-specific features as the primary surface. If your app's user-visible features are 1:1 with provider-specific API features, you've locked in.
  • Conversation history as the only memory. When you switch models, the new model's context starts empty. If history was memory, it's gone.
  • Tool schemas in vendor format throughout the codebase. Touches every integration when you switch.

How it's enforced in project-starter

The Agent App's CLAUDE.md includes a section on portable identity. Every time Claude opens an Agent App project, it sees the rule. When you ask it to design a schema or add a new feature, the rule is in context — Claude will check designs against "is this portable?" before "is this convenient?"

You can amend the rule (loosen, tighten, scope) by editing your project's CLAUDE.md.

How it relates to agent-memory-systems

Memory systems is how you implement the persistence layer (vector stores, structured DBs, chunking). Portable identity is why that persistence has to live in your control rather than in the LLM. Read them together.
