Memory system design

Patterns for how your agent remembers things between conversations. Covers three memory types from the CoALA framework (semantic, episodic, procedural), plus vector-store selection, chunking strategies, background consolidation, and memory decay.

The CoALA framework — three memory types

Most agent apps need some persistence — what the user said before, what they prefer, what the agent has learned. The CoALA (Cognitive Architectures for Language Agents) framework breaks this into three distinct types, each with different storage requirements.

Semantic memory — facts about the world

Stable, structured knowledge. "The user's preferred language is Spanish." "Account ID 12345 is owned by Jane Doe." "This project uses Postgres on Render."

Store as: structured records in a SQL database or typed key-value store. Cheap to update, exact-match retrieval.

Episodic memory — experiences

Discrete events that happened during interactions. "On 2026-05-14, the user asked about pricing and the agent failed to find the enterprise tier." "Last Tuesday, the agent corrected a calculation mistake."

Store as: vector embeddings of episode summaries in a vector database. Retrieval by semantic similarity — "what did we talk about that was like X?"

Procedural memory — how to do things

Skills, routines, learned behaviors. "When the user says 'summarize my day,' query the calendar API, then the journal, then format as a bulleted list." Procedural memory is what makes an agent get better at recurring tasks.

Store as: structured task templates, often as JSON or markdown skill files. Distinct from semantic memory because it describes processes, not facts.

Short-term vs. long-term

The CoALA types above are all long-term. Most apps also need short-term memory — the running context of the current conversation. This usually lives in the LLM's context window, padded with relevant retrievals from long-term memory.

The split matters because context window is expensive and bounded. Short-term holds the active thread; long-term holds everything else; retrieval pulls the relevant slice of long-term into short-term when needed.

Vector store selection

The skill compares the practical options:

  • Pinecone — managed, fast, expensive at scale. Good default for production apps that need someone else to operate it.
  • Qdrant — open-source, self-hostable, supports advanced filters. Good for teams that want control.
  • Weaviate — open-source, modular, supports hybrid search (vector + keyword).
  • ChromaDB — embedded, simplest dev experience. Good for prototypes; less clear at scale.
  • pgvector — Postgres extension. Right call if you already have Postgres and don't want to operate a second datastore.
  • LanceDB — open-source, embedded, optimized for AI workloads. Often a good fit for self-hosted Claude apps.

The decision: almost always either pgvector (you already have Postgres) or one of LanceDB/Chroma (no extra ops). Start with one of those; consider migrating to managed only when scale demands.

Chunking strategies

Long documents have to be broken into chunks before they can be embedded and stored. Strategy matters more than people expect.

  • Fixed-size chunking — e.g. every 500 tokens. Simple, terrible at preserving meaning across chunk boundaries.
  • Semantic chunking — break at sentence or paragraph boundaries, then group semantically related sentences. Preserves meaning; computationally heavier.
  • Structure-aware chunking — respect document structure (headings, sections, tables). Best for technical documents.
  • Contextual chunking — prepend each chunk with a brief summary of the document it came from. Improves retrieval relevance at the cost of more tokens.

Background memory formation

Don't try to store memories in the user's request path — that adds latency the user feels. Instead:

  1. During the conversation, log raw events to a queue.
  2. Asynchronously, a worker reads the queue, summarizes events into episodes, embeds them, stores in the vector store.
  3. The user's next request retrieves from already-consolidated memory.

This is called background consolidation — modeled on how human sleep consolidates short-term experiences into long-term memory.

Memory decay

Old memories that no longer matter should fade. Two approaches:

  • Time-based decay — older memories get lower retrieval scores. Eventually drop below a threshold and get pruned.
  • Utility-based decay — track which memories actually get retrieved and used. Memories that are never used drop out.

Without some form of decay, the vector store grows unboundedly and retrieval quality degrades (every query returns stale, irrelevant hits).
