Multi-agent coordination patterns

For agent apps that need more than one agent. Documents three coordination primitives (Handoff, Assign, send_message) and three orchestration shapes (supervisor/worker, peer-to-peer, orchestrator-led). The patterns are independent of any specific library; the same shapes appear in AWS's CLI Agent Orchestrator and most multi-agent frameworks.

The three coordination primitives

Handoff — synchronous task transfer

Agent A passes the active task to Agent B and waits for B to finish before continuing. Used when there's a clean baton-pass: one agent's output is the next agent's input.

Example: a research agent gathers source material, hands the conversation to a writing agent that drafts the article. The research agent doesn't see the article; the writing agent doesn't redo the research.

Assign — asynchronous spawn

Agent A spawns Agent B (or several Bs) and continues its own work in parallel. Used when sub-tasks are independent and can run concurrently.

Example: an orchestrator dispatches three research agents to investigate three sources at once. While they work, the orchestrator can summarize what it knows so far.

send_message — point-to-point communication

Agents already in flight exchange messages without blocking. Used for sharing intermediate results, coordinating constraints, or raising conditions that affect another agent's work.

Example: while writing, the writing agent realizes it needs one more source. It sends a message to the research agent rather than restarting the whole pipeline.

The three orchestration shapes

Shape 1: Supervisor / Worker (hierarchical)

One supervisor agent owns the goal. Workers are specialists the supervisor calls on. Workers don't talk to each other directly. 

            ┌──────────────┐
            │  Supervisor  │
            └──┬────┬───┬──┘
               │    │   │
       ┌───────▼─┐ ▼ ┌──▼─────┐
       │ Worker  │ │ │ Worker │
       │ (research) │ │ (write) │
       └───────────┘ │ └────────┘
                     ▼
                  Worker
                 (review)

Good for: centralized control, auditability, policy enforcement. Easy to reason about.

Watch out for: the supervisor becomes a bottleneck. All decisions flow through it.

Shape 2: Peer-to-Peer (network)

Agents talk to each other directly without a central coordinator. Each agent decides who to ask based on its goal.

Good for: emergent collaboration, no single point of failure, can scale to large agent counts.

Watch out for: hard to debug ("why did agent C ignore agent B?"), hard to enforce global constraints, race conditions on shared state.

Shape 3: Orchestrator-Led (pipeline)

A lightweight orchestrator routes work through a fixed sequence of specialists. The orchestrator doesn't decide what to do — it just knows the order.

Good for: well-known workflows where the shape is stable (CI pipelines, content publishing, data ETL).

Watch out for: not really agent behavior — it's a workflow engine that happens to use LLMs. That's fine, but call it what it is.

Conflict resolution

When two agents disagree about state, you need a policy. Three options:

  • Last writer wins — simple, often wrong for important state.
  • Supervisor arbitration — escalate to a higher-level agent. Works in hierarchical shapes.
  • Explicit lock or claim — agents claim ownership of state before modifying. Avoids conflicts but adds latency.

When you DON'T need multi-agent coordination

Most apps don't. Multi-agent shapes pay for themselves only when the split has a real reason — different personas, parallel work, or context-isolation requirements. Single-agent apps can skip this entirely.

Decision check: can you describe what the agent does in one paragraph? Then you probably need one agent, not several.

Public implementations

AWS CLI Agent Orchestrator (CAO)

AWS Labs' reference implementation. Documents all three primitives and shapes; provides a Python library. 

pip install cli-agent-orchestrator

Source: github.com/awslabs/cli-agent-orchestrator. Announcement post: AWS Open Source Blog.

muratcankoylan's multi-agent-patterns skill

Part of the Agent Skills for Context Engineering pack. 

/plugin marketplace add muratcankoylan/Agent-Skills-for-Context-Engineering
/plugin install multi-agent-patterns@context-engineering

How it relates to ai-agents-architect

ai-agents-architect helps you decide whether you need multiple agents. agent-orchestrator helps you coordinate them once you've decided you do. They're sequential, not alternatives.
