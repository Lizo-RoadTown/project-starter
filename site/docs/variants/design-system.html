Design-system structure

A pattern (and an associated skill) for organizing visual values — colors, spacing, font sizes — in three layers so they stay consistent across every page and theme. Changing one primitive value ripples through the entire UI without touching component code.

Heads-up: design-system is a generic name with multiple community skills under it. The pattern itself is universal; what differs between implementations is exactly which token names ship and which framework adapters are included. For project-starter, the pattern matters more than which specific package you install.

The three-tier token architecture

Design tokens are named values — color-blue-500, spacing-md, radius-lg. The three-tier pattern separates them by purpose, which is what makes the system maintainable.

Tier 1 — Primitive tokens

Raw values, no context. The visual vocabulary.

--color-blue-500: #1A73E8;
--color-blue-600: #1557B0;
--spacing-1: 4px;
--spacing-2: 8px;
--spacing-3: 12px;
--font-size-base: 16px;
--radius-sm: 4px;
--radius-md: 8px;

Primitives are content-free. They name nothing about where they're used.

Tier 2 — Semantic tokens

Reference primitives by purpose, not by value.

--color-action-primary: var(--color-blue-500);
--color-action-primary-hover: var(--color-blue-600);
--color-text-primary: var(--color-zinc-900);
--color-text-muted: var(--color-zinc-500);
--spacing-component-padding: var(--spacing-3);
--radius-card: var(--radius-md);

Semantic tokens encode intent. This is the layer that matters most for AI agents: when Claude has to pick a color for a "primary action button," it can read color-action-primary and know it's the right answer. Without semantic naming, Claude has to guess from raw values — which leads to inconsistent picks across files.

Tier 3 — Component tokens

Specialize per component, when needed.

--button-primary-bg: var(--color-action-primary);
--button-primary-bg-hover: var(--color-action-primary-hover);
--button-primary-padding: var(--spacing-component-padding);
--card-background: var(--color-surface-1);
--card-padding: var(--spacing-4);
--card-radius: var(--radius-card);

Component tokens are optional. Use them when a component has unique requirements (a button's padding might differ from a card's padding even though both are "component padding" in the abstract). Skip them when semantic tokens are enough.

Why three tiers (not one, not two)

The flexibility lives in the indirection:

  • Change a brand color → edit one primitive (--color-blue-500). Every semantic and component token derived from it updates automatically.
  • Switch to dark mode → swap a different semantic-token sheet that points the same names at darker primitives. Component code never changes.
  • Add a new brand or white-label → swap primitive + semantic sheets together. Components are unaware.

Why semantic naming matters for AI agents

An LLM writing new components has to pick token values. If your codebase uses primitives directly (color: var(--color-zinc-900) everywhere), the model has to infer which zinc shade is the "primary text color" by reading examples. It will guess wrong sometimes.

If your codebase uses semantic names (color: var(--color-text-primary)), the model just reads the name and picks correctly. Semantic naming is machine-readable design intent.

When it kicks in

Activates on prompts about:

  • Setting up the project's design tokens for the first time
  • Adding a new theme (dark mode, brand variant, white-label)
  • Refactoring an existing codebase to use the three-tier pattern
  • Choosing where to add a new visual concept (which tier does it belong on?)

Public implementations to consider

Two community skills publish under or near this name:

  • Triptease's claude-skill-design-system — github.com/triptease/claude-skill-design-system. Clone into .claude/skills/. Encodes the three-tier hierarchy plus component-architecture patterns (polymorphic, compound) and theme switching.
  • design-system-patterns on the MCP Market. Similar scope, slightly different opinions on naming conventions.

No single canonical Anthropic version exists under the exact name design-system. The template references the pattern; pick whichever implementation fits your project's framework.
